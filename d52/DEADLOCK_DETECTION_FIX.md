# 死锁检测修复说明

## 问题描述

原算法在处理菱形依赖（Diamond Dependency）场景时存在误报问题：
- 场景：A依赖B和C，B和C都依赖D
- 问题：算法错误地将跨分支的共享下游节点判定为死锁

## 根本原因

原DFS算法虽然有`recStack`（递归栈标记），但状态管理不够清晰。关键改进是使用**三态访问标记**替代简单的布尔标记。

## 修复方案

### 1. 三态访问状态定义

```go
type VisitState int

const (
    Unvisited VisitState = iota  // 未访问
    Visiting                     // 正在访问（在当前递归栈中）
    Visited                      // 已访问完成
)
```

### 2. 改进的DFS算法流程

```
对于每个节点：
  1. 如果状态是 Unvisited，开始DFS遍历
  2. 标记节点为 Visiting，并加入当前路径
  3. 遍历所有邻居：
     - 如果邻居是 Unvisited → 递归访问
     - 如果邻居是 Visiting → 发现真正的死锁循环（回到当前递归栈中）
     - 如果邻居是 Visited → 跳过（已处理完，不在当前栈中）
  4. 回溯：将节点从路径移除，标记为 Visited
```

## 关键改进点

**原逻辑缺陷（隐式）：**
```go
// 即使 visited[neighbor] == true
// 但 recStack[neighbor] == false 时不应该判定为循环
// 但状态转换不够清晰，容易产生混淆
```

**修复后逻辑：**
```go
// 只有当邻居状态为 Visiting（在当前递归栈中）才判定为循环
// 状态为 Visited 的节点表示已经处理完，属于其他分支的共享节点
if state[neighbor] == Visiting {
    // 真正的死锁循环
}
```

## 单元测试覆盖场景

### 无死锁场景（不应误报）
1. **菱形依赖** - A→B, A→C, B→D, C→D
2. **线性链** - A→B→C→D
3. **空图** - 无节点
4. **单节点无边** - 只有A，无出边
5. **复杂多分支共享节点** - A→B→D→E→F, A→C→D
6. **指向已访问但不在栈中的节点** - A→B→C→D, A→D

### 应检测到死锁的场景
1. **简单双向循环** - A↔B
2. **三节点循环** - A→B→C→A
3. **多个独立循环** - A↔B 和 C↔D
4. **菱形中包含循环** - A→B→D, A→C→D, D→B（形成B↔D循环）
5. **自循环** - A→A

### 牺牲者选择测试
1. **基于优先级** - 选择优先级低的
2. **基于运行时长** - 选择运行时间短的
3. **基于资源占用** - 选择资源占用小的

## 死锁检测示例

### 真实死锁（会被检测到）
```
任务A持有锁res1，等待res2
任务B持有锁res2，等待res1
边: A→B, B→A
检测结果: 循环 A -> B -> A
```

### 菱形依赖（不会误报）
```
边: A→B, A→C, B→D, C→D
DFS路径1: A→B→D（D标记为Visited）
DFS路径2: A→C→D（D状态为Visited，跳过）
结果: 无死锁
```
